**Near-miss: `cctx` bug could have killed the run.**

The `cctx` tool hardcoded all models at 200K context. During the run, `cctx` reported 60-90% when the real context was 12-19%. Two consequences:
1. The post-commit context rotation hook (kills at 65%) would have terminated the session at ~130K tokens — only 13% of actual capacity — if it had run before the director fixed `cctx`.

The director session fixed `cctx` mid-run (updated the lookup table to include 1M for `claude-opus-4-6`). After the fix, the hook correctly saw 15% and the agent's own `cctx` checks showed correct values.

**Suggestions:**
- **`cctx` should auto-detect context window.** The hardcoded lookup table is fragile. When new models or context sizes ship, `cctx` breaks silently. Options: (a) read `max_tokens` or `context_window` from the API response metadata if available, (b) infer from the model ID suffix (`[1m]` → 1M), (c) default to 1M for Opus and let the user override.
- **Recalibrate thresholds for 1M.** The percentage-based thresholds (50%/60%/65%) were designed for 200K. At 1M, 50% = 500K tokens — that's more than entire Sonnet conversations. Consider switching to absolute remaining-token thresholds: "hand off when < 200K remaining" (~20% of 1M) rather than "hand off at 50%."
- **Auto-implement model should default to Opus 1M.** Given the dramatic improvement (0 slices in 5 min vs 3 slices in 35 min), Sonnet 200K is no longer viable for auto-implement. The CLI should default to `claude-opus-4-6[1m]`.
- **Director monitoring UX needs work.** The director session spent significant turns polling with `sleep` + `tail`. `auto-implement watch` and `auto-implement stream` exist but weren't tested. A future retro should evaluate whether the observability tools give the director adequate real-time visibility.
