## What's Pending

These are **active bugs** reported by the user during testing. They should be the priority.

### 1. Apply rule during sync — emails disappear + spinner stuck

**What the user reported:**
- Clicking Apply while sync is running causes the spinner to keep spinning indefinitely
- At least 2-3 times, the email entered in the tag input **disappeared** after Apply
- The rule application seemed stuck until sync completed

**Investigation notes:**
- The Apply spinner is controlled by `applyingIds` state in `handleApplyRule` (`fathom-config-modal.tsx:~1135`). It's cleared after `updateRuleAction` resolves.
- `updateRuleAction` calls `PUT /api/fathom/{projectId}/rules/{ruleId}` on the Python API — this endpoint (`fathom.py:1649`) is `async def` but makes synchronous Supabase calls. If the event loop is still blocked by something, this would hang.
- BUT sync and evaluation now run in threads, so the event loop should be free. The `update_rule` endpoint's sync calls (~2-5ms each) shouldn't block noticeably.
- **Possible cause of disappearing emails**: The 3s sync poll calls `fetchRules()` which overwrites the local `rules` state. If the user has typed emails and not yet clicked Apply (dirty state), the poll would reset those emails to the server-side values (empty). The `EmailTagInput` component receives `emails` from the parent's `rules` state, so an overwrite would wipe the UI.
- **Possible cause of stuck spinner**: The `handleApplyRule` function finds the rule via `rules.find(r => r.id === ruleId)`. If `rules` was just overwritten by a `fetchRules()` poll, the rule's `group_a_emails` might be the old (empty) value, not what the user typed. The Apply would save empty emails, then evaluation would find 0 matches.
- **Root fix direction**: `fetchRules()` should NOT overwrite rules that have unsaved local changes (dirty state). Check `dirtyIds` before overwriting, or merge server data with local edits.

**Key files:**
- `nextjs-app/app/projects/[projectId]/config/fathom-config-modal.tsx` — `PeopleRuleEditor`, `handleApplyRule`, `fetchRules`, polling effects
- `python-services/agent_api/api/fathom.py` — `update_rule` endpoint

### 2. Stale rule match count (31 at top, 6 next to rule)

**What the user reported:**
- Applied rule during sync, matched 6 meetings
- After sync completed, saw "31 matched meetings" in the meetings list at the top
- But the per-rule match count next to the rule still showed "6" for a while
- Eventually it updated to 31 (without user action)

**Investigation notes:**
- Match counts come from `get_rule_match_counts` RPC called inside `list_rules` endpoint (`fathom.py:1626`)
- `fetchRules()` is now called in the 3s sync poll and 10s background poll (fix `0ba176304`), so counts should update. But the user still saw staleness.
- **Possible timing issue**: Post-sync evaluation runs in the sync thread. `sync_status` is cleared in the `finally` block AFTER evaluation. But the 3s poll might have fetched rules at a moment when meetings were updated but evaluation hadn't finished writing all `meeting_rule_evaluations` rows yet.
- **Possible fix**: Add evaluation progress detection — when `pending_evaluation_total` is set but `pending_evaluation_done < total`, the frontend knows evaluation is in progress and should poll more aggressively for rules. Or simply: after detecting sync completion, delay the rules fetch by a second to let evaluation finish writing.
- This bug may partially overlap with bug #1 — if the polling overwrites dirty rules, and the user has to re-apply, the count stays at the old value until the next evaluation.

**Key files:**
- `nextjs-app/app/projects/[projectId]/config/fathom-config-modal.tsx` — polling effects, `RuleMatchCount` component
- `python-services/agent_api/api/fathom.py` — `list_rules` endpoint, `_run_evaluation_sync`
