# Ideator 02 — Systems Thinker

## Priming (one sentence)
Pick the right primitive and the race condition turns into a textbook problem.

## Ideas

- **Branch-per-agent + push-to-main fanout**: each Gin session works on `gin/<session-id>` worktree, autosync pushes that branch, a server-side hook (or local merge daemon) fast-forwards `main` only when the branch's diff lints clean against committed `main`. Why: removes the shared-mutable working tree entirely; parallelism becomes a merge problem, which git already solves.
- **Per-agent git worktrees as the default**: `dx session start` materializes a worktree under `.gin/worktrees/<sid>/` sharing one objectdb. autosync pushes from there. Why: physical isolation of working tree state — the textbook fix for "shared mutable directory."
- **Two-phase commit on push (prepare/commit)**: autosync renames to `prepare-push` (rev-parse HEAD, fetch, rebase, lint diff-only, push), only on success does it advance the local ref; on failure it stashes-with-name to `gin/abandoned/<sid>/<ts>` instead of `reset HEAD~1`. Why: 2PC is the canonical primitive for "do A and B atomically across systems."
- **Append-only commit log, never reset**: forbid `reset HEAD~1` in autosync. Replace with `git update-ref refs/gin/parked/<sid>/<seq> HEAD` then `git reset --soft HEAD~1` only if recoverable ref written first. Why: log-structured storage solves data loss — never destroy, always append a tombstone.
- **MVCC for the working tree via overlayfs**: each agent sees its own copy-on-write overlay of the repo; merges happen at commit-emit time, not at edit time. Why: MVCC is how Postgres avoids readers blocking writers; same shape applies here.
- **Lock-free queue for commits**: one `gin-commitd` daemon owns push to `main`. Agents enqueue `(sha, branch, intent)` to a local socket; daemon serializes, lints diff-only, force-fast-forwards or rejects. Why: single-writer-multiple-reader is the classic concurrency primitive; eliminates push races by construction.
- **Leader-election for autosync**: only one agent's autosync runs per checkout at a time, chosen by Lamport timestamp on `.gin/leader` lockfile. Others batch their commits to the leader's queue. Why: leader-election turns N concurrent writers into one logical writer.
- **Diff-scoped pre-push hook (the principled fix)**: pre-push runs `tsgo --noEmit` and tests *only on files in the push range* (`git diff --name-only origin/main..HEAD`), not the working tree. Why: the gate should test the artifact being pushed, not ambient state. This is what CI does — local should match.
- **Pre-push runs in a clean ephemeral checkout**: hook does `git worktree add /tmp/prepush-<sha> <sha>` and lints/tests there, ignoring working tree dirt entirely. Why: the artifact under test must equal the artifact being pushed; reproducible-build hygiene.
- **Content-addressed staging area**: `dx commit` stages by writing blobs to `.gin/staging/<content-hash>` then atomically swaps refs. No `git add` of working-tree paths that another agent might mutate mid-stage. Why: content-addressing eliminates the file-path race entirely.
- **CRDT-style commit message merge**: when two agents commit to overlapping paths, the message and stat blocks are merged via a CRDT (per-author lines). Why: collaborative editors solved this; we're just doing it on commit metadata.
- **Vector-clock attribution**: every file edit carries `(session-id, lamport-time)` xattr; commits read xattrs and split into per-author commits automatically. Why: vector clocks were invented exactly to disentangle concurrent causal histories.
- **Per-agent index file**: `GIT_INDEX_FILE=.git/index-<sid>` so each agent has its own index. `git add`/`git commit` from that agent never sees another's staged set. Why: git already has the primitive (env var); we just need to use it. Removes Mode-1 collisions by construction.
- **Outbox pattern for pushes**: agents commit to local refs only; a separate `outbox` worker drains commits to `origin/main` with retry+backoff and dead-letter to `refs/gin/dlq/<sid>`. Why: outbox decouples local progress from remote availability — same as service-mesh outbox.
- **Backpressure on pre-push contention**: when push fails N times in M minutes, autosync escalates to "storm mode" — pushes go to `gin/storm/<sid>` side-branch, surfaces a `dx storm-status` for the human. Why: backpressure + circuit-breaker is how distributed systems shed load instead of cascading failure.
- **Eventual consistency via sync points**: agents push freely to per-agent refs; a lightweight reconciler (`dx reconcile`) walks all `refs/gin/*/main` and produces a unified `main` via octopus-merge or topological replay. Why: dynamo/cassandra-style — accept divergence, reconcile explicitly.
- **WAL-style commit journal**: every staged file write goes through `.gin/wal/<seq>.json` (path, blob-sha, intent, sid) before touching the working tree. Recovery on crash/reset replays the WAL. Why: write-ahead log is the recovery primitive; reflog isn't enough because it doesn't cover stash/reset interactions.
- **Hash-chain for stash naming**: stashes named `gin/<sid>/<parent-sha>/<intent-hash>` form a chain; `dx unstash` walks the chain to recover all 27 stashes deterministically. Why: hash-chain = greppable, idempotent, immutable history. No more "which stash was mine?"
- **Read-only quorum on push**: a push to `main` requires a quorum of agents to ack "my working tree is clean for the paths in this push." Other agents respond automatically based on their per-agent index. Why: Paxos/Raft pattern — coordinated commit instead of optimistic-then-blame.
- **Optimistic concurrency with versioned refs**: each agent records `seen-main = <sha>` at intent-time; push fails if `origin/main != seen-main`, autosync responds with rebase-and-retry-with-fresh-sha or queue. Why: OCC is how databases avoid locks; we already have refs as version vectors — use them.
- **Sharded ownership of paths**: declare path-prefixes belong to a session at `dx claim`. Other sessions can read but not write until released. Lints & pushes ignore non-claimed paths. Why: sharding turns shared-mutable into per-shard-mutable; lock-free reads still work.
- **CAS (compare-and-swap) for autosync**: autosync's commit-then-push becomes one CAS loop: read `origin/main`, compute new commit on top, push with `--force-with-lease=origin/main:<seen-sha>`, on fail rebase and retry. Never `reset HEAD~1`. Why: CAS is the universal primitive for safe concurrent updates.
- **Event-sourced session state**: instead of storing "current working tree," store an append-only log of `(sid, edit, ts)` events; the working tree is a projection. Recovery = replay. Why: event sourcing makes "lost work" structurally impossible — the events are the truth, files are derived.
- **Storm detector via load average on push failures**: a sliding-window over `git push` exit codes; >P% failure rate flips a circuit-breaker that defers non-critical pushes (zettels, docs) and alerts the human. Why: circuit-breakers are how Netflix survives downstream outages; same shape here.
- **Rendezvous-hashing for next-zettel-id**: replace `nextId()` filesystem scan with a content-addressed allocator: `id = hash(session-id + intent-payload) mod 100000` then collision-check against committed set. Why: rendezvous-hashing was built for "N writers picking unique slots without coordination" — the z038 race in textbook form.
- **Distributed sequence service via git refs**: `dx zettel id` does `git update-ref refs/gin/zettel-counter <new>` with `--create-reflog` and CAS via `--ref-spec`. Refs ARE a coordination service git ships with. Why: re-use existing primitives instead of inventing.
- **Reflog-as-recovery API**: `dx undo-autosync` reads `HEAD@{n}` reflog, identifies the last `reset HEAD~1` from autosync (signature pattern), and offers to recover the lost commit by cherry-pick. Why: reflog is already the WAL; we just need a UX around it. Findability without a new system.
- **Saga pattern for multi-step pushes**: a push that fails mid-way (commit, rebase, lint, push) leaves compensating actions in `.gin/saga/<id>/` that `dx recover` can apply. Why: sagas handle distributed-transaction failure by compensation, not rollback — perfect fit because git's "rollback" (reset) is what's eating our work.
- **NATS-jetstream-style commit stream**: agents publish commits to a local jetstream subject; a single subscriber pushes to `main` with at-least-once delivery and dedup-by-sha. Why: messaging systems solved exactly-once-effective delivery decades ago; commits are messages.
- **Linearizability via fence tokens**: each `dx push` gets a monotonic fence token; the push proceeds only if `origin/main`'s fence token < ours. Resolves the autosync-races-itself case. Why: fencing tokens prevent zombie writers — exactly the autosync-on-resumed-session bug.
- **Read-your-writes session consistency**: every Gin session caches its own committed shas; if `git log origin/main` doesn't contain them after N seconds, it raises (not silently retries) — surfacing Mode-2 collisions immediately. Why: session-consistency guarantees are how Cosmos/Dynamo expose anomaly classes the user can act on.
- **Tombstone-and-revive for "lost" commits**: any `reset HEAD~1` writes `refs/gin/tombstones/<sha>` first, then refuses if writing fails. `dx revive <prefix>` lists tombstones and one-command-restores. Why: soft-delete is universal for "never lose data even when the user asks you to."
- **Gossip protocol between sibling Gins**: agents announce their working-tree path-claims via a local socket; before staging a path, an agent checks the gossip table. Why: gossip is how distributed systems share state without coordination overhead — fits our N-parallel-Gin shape.
- **Causal-broadcast of "tree dirty" events**: when an agent edits a file, it broadcasts `dirty(<path>, <sid>)`; other agents' pre-push hooks ignore those paths' lint failures. Why: causal broadcast lets each agent locally compute "is this dirt mine or theirs?" — converts blocking into routing.
