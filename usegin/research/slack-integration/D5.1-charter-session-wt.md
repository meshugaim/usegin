## What this is

A `dx session-wt` primitive that gives every autonomous Claude session its
own git worktree under `.worktrees/<session-id>/`, so siblings stop
colliding through the shared `.git/index` and shared working tree.

End-state: a Claude session, on its first commit-ish action, lands itself
in a peer worktree, operates entirely there, pushes `HEAD:main` from the
worktree, and tears down on session end. The shared root
(`/workspaces/test-mvp/`) becomes the human's lane — never an agent's.

Why: 9 distinct cluster touches since slice-1 era (z038), 6 incidents in
24h after the prior tikur. Every band-aid (autosync stash, attribution
notes, snapshot tripwire) reduces blast radius; only worktree-isolation
removes the race.

## The build

Three pieces. None individually large; the integration is the work.

### 1. CLI: `dx session-wt`

New command in `tools/dx/src/commands/session-wt.ts`.

```bash
dx session-wt enter [--session-id <id>] [--branch <name>]
# Creates .worktrees/<id>/ if missing; outputs `cd <path>` line; idempotent.

dx session-wt exit [--session-id <id>] [--keep]
# Rebases the worktree's commits onto origin/main, pushes, removes
# worktree (unless --keep). Refuses if uncommitted changes exist.

dx session-wt status
# Lists active session worktrees with branch + ahead/behind vs main.

dx session-wt gc [--older-than 7d]
# Removes orphaned worktrees whose session is no longer running.
```

Branch-naming convention: `gin/<session-id-short>` — distinct per session,
all push to `main`. Autosync already supports this (line 112 of
`autosync.ts`).

### 2. SessionStart hook

Add to `.claude/settings.json` SessionStart hook chain. On every session
start in `/workspaces/test-mvp/`:

1. Read `$CLAUDE_SESSION_ID` (the running session's id).
2. Run `dx session-wt enter --session-id $CLAUDE_SESSION_ID`.
3. The hook outputs `additionalContext` telling the agent
   *"You are in a per-session worktree at `.worktrees/<id>/`. All commits
   and edits should happen here, not in the parent."*
4. Working directory of subsequent tool invocations: the worktree path.

The hook is **opt-in** via env var or settings flag — not on for everyone
by default, because human-driven sessions don't need it. Default-on for
sessions launched by the autonomous-vibe protocol (z091).

### 3. Teardown

On `Stop` hook (or explicit `dx session-wt exit`):

1. If clean: rebase onto origin/main, push, remove worktree.
2. If dirty: leave worktree intact, log a warning, surface in
   `dx session-wt status`. Operator decides whether to recover or `gc`.

## Test plan (sketch)

- Unit: `session-wt enter` is idempotent; refuses `enter` when
  `.worktrees/<id>/` exists with uncommitted changes; rebase-on-exit.
- Integration: spawn 3 SDK sub-agents in parallel, each writing distinct
  files; assert all 3 commits land on origin/main with correct payloads
  and messages, no `staging-drift` tripwire fires.
- Empirical: re-run the autonomous-vibe protocol with parallel batches
  enabled (was paused per z099 §1 / D5); zero attribution swaps in the
  next 24h is the success criterion.

## Cost / risk

- **Cost:** ~half-day to build the CLI + hook + tests; ~50–100MB disk
  per active session (linked, but real on `node_modules` etc.).
- **Reversible:** flag-gated. If something breaks, set the flag off and
  fall back to shared root.
- **Risk:** agents producing absolute-path artifacts that don't transplant.
  Mitigation: existing CLAUDE.md repo-relative-path conventions
  (already mostly enforced).
- **Risk 2:** nested invocations inside the worktree (e.g. another
  `claude` instance) re-entering — mitigation: the `enter` command is
  idempotent and checks `$CLAUDE_SESSION_ID` to be the same.

When you (Lihu) `claude` in this repo, do you want the same worktree
auto-allocation, or do you want to stay in `/workspaces/test-mvp/` and

- **(b) Off for me, on for autonomous.** You stay in shared root; only
  autonomous-vibe spawns + sub-agent SDK runs go to worktrees.

agents that need isolation declare it; humans are unaffected unless they
opt in. It also lets us roll out gradually — turn it on for one
autonomous run, watch for issues, then expand.

### Q2 — Where do worktrees live?

- **(a) `.worktrees/<session-id>/`** under the main repo (current
  pattern, gitignored). Pros: locality, easy `ls`. Cons: nests inside the
  repo; some tools choke on nested git dirs (rare).
- **(b) `~/.gin-worktrees/<session-id>/`** outside the repo. Pros:
  cleaner repo; pros for absolute-path-allergic tools. Cons: harder to
  discover; needs more cross-env coordination (Gitpod, codespaces).

**Lean: (a)** — matches `.worktrees/eng-1039` / `eng-1041` already
present; minimum new conventions; gitignore already covers it.

### Q3 — How aggressive on teardown?

- **(a) Auto-rebase + push + remove on Stop.** Most cleanup, zero manual
  steps. Risk: a session that errored mid-task loses its worktree state
  before the operator sees it.
- **(b) Auto-rebase + push, manual `gc`.** Worktrees accumulate; weekly
  `dx session-wt gc --older-than 7d` cleans them. Compromise.
- **(c) Manual everything.** `dx session-wt exit` is operator-explicit;
  `gc` is too. Maximum safety, more friction.

**Lean: (b).** The gc cycle is short enough that disk doesn't blow up;
operators can always run `gc` themselves; failed sessions leave evidence.
