## Code quality

**GREEN.**

**Helper** (`nextjs-app/lib/slack-token-decrypt.ts`, 44 lines): pure-functional,
no `server-only`, no `next/headers`, no Sentry — three imports total
(`decrypt`, `isEncrypted`, type `SlackTokenDecryptResult`). Discriminated-union
return is the right shape: `{ ok: true, token } | { ok: false, reason:
"legacy_raw" | "decrypt_failed", error: unknown }`. Caller sees exactly the
information needed to differentiate the two failure modes.

The **legacy_raw vs decrypt_failed split is valuable, not noise.** They
correspond to two distinct ops events: (i) "we have a row that pre-dates
encrypt-on-write — backfill missed it" (audit-trail, never expected in
greenfield); (ii) "we have an encrypted row but key/AAD/ciphertext is wrong"
(actual decrypt failure — wrong key, rotation, tampering). Mixing them
would lose the backfill-leak signal at scale. Sentry tagging at
`nextjs-app/app/actions/project-slack.ts:363-370` (legacy_raw → `captureMessage`
with `level: "error"` and a descriptive `"backfill leak"` string) and
`L381-384` (decrypt_failed → `captureException(decryptResult.error, ...)`)
distinguishes them on the dashboard side too.

**Latent thing, not a bug:** the `error: unknown` on the failure branch
hurts a tiny bit at the call site — `Sentry.captureException(decryptResult.error, ...)`
accepts `unknown` so it works, but a future reader who wants `error.message`
would have to narrow. Not worth fixing — `unknown` is the honest type for a
caught throwable, and the only consumer today is Sentry which doesn't need
narrowing. Worth recording as a shape note only.

**Action site** (`project-slack.ts:343-391`): clean three-branch handling.
The decrypt is between the install-status check (L324-341) and the Slack
HTTP fetch (L398+) — in the right place. The error strings are user-facing
and consistent: both reasons surface "Reconnect Slack to refresh the
credential." which is the actionable instruction. No extra surface — no
unused result fields, no dead branches, no defensive reads.

**Tests** (`nextjs-app/tests/unit/lib/slack-token-decrypt.test.ts`, 116 lines,
5 tests, 12 expects, all green — verified): meaningful, not vacuous.
- happy path → encrypts then decrypts with same AAD; asserts `token === PLAINTEXT`.
- legacy_raw → asserts a raw `xoxb-` value is rejected with `reason: "legacy_raw"`.
- AAD mismatch → encrypts under a *different* `rowId`; asserts `decrypt_failed`
  + `error` is truthy. This is the test that proves AAD is actually being
  verified, not just being passed.
- tampered ciphertext → flips a byte in the third wire segment; asserts
  `decrypt_failed`. This proves auth-tag enforcement.
- missing env var → `delete process.env.TOKEN_ENCRYPTION_KEY`; asserts
  `decrypt_failed` (loadKey throws). Quality-gate per z091.

The env var test correctly captures and restores `ORIGINAL_KEY` in
`afterEach` per the `tests/CLAUDE.md` rule "Set `process.env` in `beforeEach`
without cleanup → add `afterEach`." Conventions held.

The deletion of `nextjs-app/tests/unit/actions/list-slack-channels-decrypt.test.ts`
in `222cf2d02` (-223 lines) was the right call — that file was unreachable
in the full suite due to sibling `mock.module` of `@/app/actions/project-slack`
in the C3 component test. The lib-extract makes those branches reachable
*and* keeps the action file thin (+0 net test-target bytes, since the action
test would have been carrying the same 5 cases plus mock-setup boilerplate).

1. **Drop a zettel naming the mock-leak / action-test-reach pattern.** Third

   file that's `mock.module`-substituted by sibling component tests cannot
   host its own internal unit tests in the same test run. Extract pure
   logic to `lib/` and test there." Cite z095/z096/this turn as the chain.
   Same-turn placement (per `first place we looked` rule) — the natural
   address is `usegin/zettel/zettels/z098-*.md`. Optionally cross-link
   from `nextjs-app/tests/CLAUDE.md`'s "Exception: server actions in
   shared mocks" subsection so a future Gin reading the mock rules sees
   the consequence.

2. **Recommended next slice: D4 (`#usegin` outbox channel default + ENG-ID
   auto-link).** Reasoning, weighing the four candidates the orchestrator
   listed:
   - **C4 (Events API ingestion)** is the highest customer-leverage but
     too big for a single-agent slice with the autosync fight unfixed —
     ingestion route + sync worker + lifecycle handlers is multi-component
     and the kind of work the previous parallel-batch was doing when the
     z094/z095/z096 cluster fired. Risk-asymmetric right now.
   - **C5 (lifecycle handlers only)** is a defensible fallback if D4
     feels too small — but it's still a fragment of C4 and would split the
     ingestion work into harder-to-merge pieces.
   - **D4** is small, self-contained to `tools/dx/`, gives Lihu a visible
     win on the demo path (the `dx slack send` flow already works with a
     real token; default `#usegin` + auto-link is a polish that compounds

     under the autosync fight than the C-track.
   - **Marketplace `[LIHU UNKNOWN]` polish** is also single-agent-safe
     but is a doc/security-questionnaire fill rather than a build slice;
     it's lower energy and could be a 30-minute side-pick after D4.
   Best single-agent slice = **D4**. C4 should wait on the z097 autosync
   resolution before fanning out parallel.

   Z097 + the FOR-TOM #8/9/10 chain are unresolved; the next parallel
