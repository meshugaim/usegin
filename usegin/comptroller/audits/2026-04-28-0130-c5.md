### `lib/slack-event-signature.ts` (95 lines, 15 tests)

The HMAC verifier is **correct in every detail I checked:**

- **Constant-time compare**: `crypto.timingSafeEqual` (line 108) — the
  one canonical correct primitive.
- **Length-mismatch guard**: line 105-107 explicitly checks
  `provided.length !== expected.length` BEFORE the timingSafeEqual call,
  because timingSafeEqual *throws* on mismatched lengths. Tested
  explicitly at `slack-event-signature.test.ts:146-155`. This is the
  exact bug-shape that catches inexperienced HMAC implementations.
- **Replay window symmetry**: stale (line 79-81) + future (line 82-86),
  both ±5min, both tested at `signature.test.ts:73-97`. Future-dated
  rejection comment ("clock skew, or someone trying to bank a signature
  for later") is doctrinally correct — an attacker who captures a future
  signature could replay it in 4-1/2 minutes.
- **`v0=` prefix check + hex regex**: line 88-94 split into two distinct
  reasons (`bad_signature_format` covers both, but the test at L99-119
  covers each branch separately). Good because if Slack ever ships a
  v1, the prefix check is the boundary.
- **Discriminated VerifyResult**: every failure mode named distinctly
  (`missing_secret` / `missing_headers` / `stale_timestamp` /
  `future_timestamp` / `bad_signature_format` / `signature_mismatch`).
  This is what the route turns into Sentry `reason` tags. Excellent for
  separable production diagnostics.
- **`signSlackEvent` helper**: line 120-130. Test-only + debug. The doc
  explicitly says "don't ship code that signs *outbound*" — that's the
  right boundary; outbound signing would mean we're acting as Slack,
  which we never do.
- **No env reads**: all inputs explicit (`signingSecret`, `nowSeconds`).
  Pure function. Testable without process.env mutation.

**One latent shape, not a bug:** the timestamp parser at L72-75 rejects
non-integer timestamps with `reason: "missing_headers"` — semantically
that's a *bad-format* failure, not a *missing* one. Doesn't affect
security (still rejects), but if someone in 6 months greps Sentry for
`reason=missing_headers` they'll see "looked-like-junk timestamps"
mixed in with literal-missing. **Note, not fix.**

### `lib/slack-event-handlers.ts` (266 lines, 9 tests)

**Three handlers + dispatcher.** All three follow the same shape: scope
by `(team_id, app_id, enterprise_id NULL-vs-present)`, find install ids,
return early with `no_install_match` if none, else apply the
side-effect, return structured `ApplyResult`.

- **`enterprise_id` NULL handling correct.** `selectInstallScope`
  (L57-74) builds the query with `.eq("slack_enterprise_id", enterpriseId)`
  when present, `.is("slack_enterprise_id", null)` when absent. Tested
  both directions at `handlers.test.ts:159-186`. Matches the migration's
  split partial unique indexes (per the C5 commit message — I verified
  the migration file `20260427182028_slack_installs.sql` exists and the
  schema status enum at L71 is `('active', 'token_revoked', 'error')`,
  which means `status: "token_revoked"` written by the handlers (L114,
  L182) is a valid value).
- **`app_uninstalled` semantics:** mark all installs revoked + DELETE
  bindings. Correct — the bot is gone, the bindings can't ingest, no
  reason to keep them.
- **`tokens_revoked` semantics:** mark install revoked, **PRESERVE
  bindings.** Note `bindings_preserved_for_reconnect` is appended
  (L190). Correct — admin invalidated the token but the customer may
  reconnect the same workspace; their channel bindings are the
  customer's choices and shouldn't vanish on a token rotation.
- **`channel_rename` STRICT-BREAK:** delete bindings where
  `slack_install_id IN (...) AND slack_channel_id = <renamed_id>`
  (L255-259). This is the CF9 RLS-leak vector — a `#hr-only` →
  `#general` rename would silently re-point our indexer at a public
  channel; we must refuse to follow. Test at
  `handlers.test.ts:209-225` confirms `slack_channel_id` is in the
  delete filter. The note `renamed_to:<name>` is appended for audit.
- **Idempotency:** Slack retries non-200 events; our handlers are
  effectively idempotent because UPDATE-to-revoked is monotonic
  (re-revoking a revoked install is a no-op-shaped row write) and
  DELETE-bindings is naturally idempotent (delete of zero rows is
  fine, returns count=0). The `no_install_match` early-return path
  (L98-109, L166-177, L242-253) means even a totally-spurious replay
  doesn't write anything.
- **Error reporting:** when an UPDATE or DELETE returns an error, the
  handler **does not throw** — it appends `{op}_error:{message}` to
  notes (L119, L127, L187, L261). The route then logs notes to Sentry
  `extra`. This is the right shape: handlers are tolerant, the route
  decides whether to surface as `applied` (with notes) or
  `handler_threw`. **A failing UPDATE on installs followed by a
  successful DELETE on bindings would still report `applied`** — that's
  the orchestrator's call (do we 200 or 500?), and 200 + Sentry note is
  defensible because Slack will retry on 5xx and the partial state
  isn't safer to retry. Worth knowing the shape; not a bug.
- **Type-system tautology defense:** `handleChannelRename` at L219-231
  has a runtime check `if (envelope.event.type !== "channel_rename")`
  even though the dispatch flow type-narrows to it. Defensive in case
  someone calls the handler directly. Cheap.

### `app/api/slack/events/route.ts` (171 lines, 8 tests)

- **Raw-body-first flow.** Line 59 `await request.text()` — *not* a
  JSON roundtrip. Then signature verify (L64-69), then JSON.parse
  (L84). This is the **non-negotiable order** for Slack events: HMAC
  is over the raw bytes. Comment at L7-8 spells it out. Test at
  `route.test.ts:155-163` confirms invalid signature = 401 with no body
  trust (uses fake `v0=00...0` 32-char sig — passes the `bad_signature_format`
  hex regex but fails the actual HMAC compare because the length
  doesn't match expected 64 hex; landing in `signature_mismatch`).
- **Always-200 for `event_callback` after verification.** Even handler
  exceptions (`handler_threw`, L143-145) and missing admin client
  (`admin_unavailable`, L122-127) and unhandled inner-types (L148-158)
  return 200. Slack retries indefinitely on 5xx, and our handlers
  aren't safe to re-run if they partially-succeeded — better to log
  and move on. Comment at L141-142 names the rationale.
- **Pre-200 401s are correct.** `signature_mismatch` / `stale_timestamp`
  / `missing_secret` / `missing_headers` all return 401 (L77-80). Bad
  JSON (post-signature) returns 400 (L89). These are *not* event-callback
  retries; they're pre-flight rejections that an attacker could fire,
  and Slack only retries actual event callbacks.
- **URL verification challenge handling**: L93-99. Returns the
  challenge string verbatim with `content-type: text/plain`. Tested
  at `route.test.ts:133-139`. The DEMO.md note at L142 mentions Slack
  pings this once when saving the events URL — correct.
- **Sentry tag distinctness:** I enumerated every tag emission and
  every `event_outcome` value:

  | Outcome value           | Where             | What |
  |-------------------------|-------------------|------|
  | `rejected`              | L75 (signature)   | 401, with `reason` sub-tag |
  | `bad_json`              | L87               | 400 |
  | `unknown_envelope`      | L108              | 200, non-event_callback |
  | `admin_unavailable`     | L124              | 200, deferred |
  | `handler_threw`         | L136              | 200, with `event_type` |
  | `unhandled`             | L153              | 200, dispatch returned null |
  | `applied`               | L164              | 200, normal success |

  All seven are distinct. **A Sentry filter on `event_outcome` will
  separate every production failure mode cleanly.** This is the
  cleanest receiver-tag layout I've seen in this codebase.

- **One small Sentry-shape note:** the `applied` event tags include
  `team_id` and `app_id` as **tag values** (L166-167). Sentry tag
  values are *indexed* and have a max-cardinality limit (5000
  distinct values per tag in some plans). One per AskEffi customer
  workspace = OK at our scale, but if we ever hit thousands of
  customer Slack installs, those tags should move to `extra`. **Not
  a bug today; a known shape for future scale.**
- **`EventCallbackBody extends SlackEventEnvelope` with empty body
  (L52-54).** Dead shape — `extends X { /* nothing */ }` is a no-op
  alias. Could be `type EventCallbackBody = SlackEventEnvelope`. Mild
  cosmetic dent; nothing else.

### Tests

**`signature.test.ts` (15 tests):** every reason value covered,
deterministic-output check, body-tampering check, wrong-secret check,
length-mismatch guard. **No vacuous tests** — every assertion is
load-bearing on the discriminated result.

**`handlers.test.ts` (9 tests):** clever Supabase chain-builder mock
(`buildAdminMock`, L30-122) that records query shape — lets the test
assert *what was queried*, not just *what was returned*. Tests cover:
happy path per handler, no-match early return, enterprise_id NULL vs
present scope, dispatcher routing, dispatcher null for unhandled
type. **Mock is hand-rolled, no `mock.module` for Supabase** — exactly
the z098 lib-extract pattern Yohai-3/audit-3 codified. Top-of-file
`mock.module("next/headers", …)` is the standard tests/CLAUDE.md
shape for any test that touches a route module ancestor.

**`route.test.ts` (8 tests):** URL verification challenge round-trip,
event_callback dispatch, invalid signature → 401, stale timestamp →
401, missing secret → 401, bad JSON → 400, unhandled inner type → 200
unhandled, non-event_callback envelope → 200 ignored. The
`beforeEach` at L113-123 sets up the env var + spies the admin client
*before* importing the route — that's the correct order, because the
route reads `process.env.SLACK_SIGNING_SECRET` lazily but the admin
spy must be in place before the import resolves. Subtle and right.
The `afterEach` (L125-131) restores the original env var
deterministically.

**Vacuous-test check:** none. Each test asserts on response status,
response body shape, or a property of the result. All load-bearing.

**One soft note:** the route test uses `NOW = () => Math.floor(Date.now()/1000)`
and signs with the live clock (L34). The route also uses live clock
inside the verifier. Theoretically there's a 1-second-window race
where the test signs at second N but the verifier reads at second N+1,
crossing a boundary that matters for the stale-timestamp test
(`signingTimestamp: NOW() - 60 * 60`). In practice, 60-minute drift
swallows any such second-edge — the test is robust. Worth knowing
shape only.
