## Code quality

**GREEN, with one minor shape note.**

**The regex** (`links.ts:38`):

```js
const ENG_ID_PATTERN = /(?<![\w/|<])ENG-(\d+)(?!\w)/g;
```

I traced this against 16 edge cases (a one-off Bun probe — the test file
covers 9). All correct:

- `"ENG-123."` / `"(ENG-123)"` / `"[ENG-123]"` → wraps cleanly, punctuation
  preserved.
- `"ENG-123-followup"` → wraps `ENG-123` only (good — `-` isn't `\w`, and
  `-followup` is a hyphen-suffixed extension, not a longer ID).
- `"_ENG-1"` → not wrapped (lookbehind catches `_` as `\w`). Defensible —
  `_ENG-1` would not be a Linear ID in any context.
- `"engENG-1"` → not wrapped (good — continuation of a word).
- `"<ENG-1>"` → not wrapped (the `<` lookbehind, defensive against the
  function's own output shape).
- `"ENG-1 ENG-2"` → both wrap. Adjacent IDs work.
- Already-wrapped + adjacent unwrapped:
  `"<…|ENG-99> ENG-100 after"` → `ENG-99` skipped (lookbehind on `|`),
  `ENG-100` wrapped. **This is the test that matters most** — confirms the
  function is genuinely idempotent against its own output, not just
  against fully-clean text. The repo test
  (`links.test.ts:33-37`) covers run-twice idempotency on a single ID; my
  probe confirms the harder mixed case.

**Latent shape, not a bug:** the lookbehind treats `_` as a "longer
identifier" prefix. If Linear ever emits issue IDs with underscore prefixes
in templated text (it doesn't, but worth noting), this regex would skip
them. **Not a fix; a known shape.** No action needed; if it ever bites it
will be obvious.
