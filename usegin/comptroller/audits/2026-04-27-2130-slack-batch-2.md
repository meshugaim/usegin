## Fight signal

**RED.** This is the dominant finding.

**The fight is autosync's stage-everything-staged behavior in a multi-agent monorepo.** Three distinct failure modes have now fired in a single batch:

1. **Mode-1 attribution swap** (z096): an agent's `git add; git commit -m "X"` lands its files under a sibling's "Y" message. Three swaps inside ENG-5413 alone. Detection: `git show HEAD --stat | grep <my-files>`. No prevention.
2. **Reset-wipe of untracked files** (z094): one agent's `git reset --hard` (e.g. autosync companion undoing a stray commit) blasts another agent's freshly-Written-but-not-yet-staged files. C3's UI files are the casualty visible on origin/main right now.
3. **Cross-agent pre-push block** (z095): pre-push runs tsgo against the working tree, not the commit range. Agent A's clean commit gets blocked by sibling agent B's half-imported component. Marketplace and ENG-5413 both hit this.

The agents are working *around* this, not *with* it: writing follow-up commits to re-attribute their work (z096), grepping payloads instead of trusting log messages, fixing unrelated lint blockers as the price of admission to push. Per `feedback_friction_loop`, this is the moment to STOP and raise the constraint, not lower it. The four open-question proposals exist (z094 stash-before-reset; z095 (a) shrink autosync window / (b) range-scoped pre-push / (c) `dx wait-for-clean-tree`; z096 stage-or-skip) but no path is chosen.

This is also why C3 is broken. The reset-wipe in z094 is not a one-off "we'll be more careful next time" — it's a structural property of running N autonomous agents over one checkout with an autosync companion. **Any new parallel batch is exposed to the same wipe.**

There is also a smaller, separate fight: **the Slack app-registration gating** (FOR-TOM.md item 1–2). The demo target requires real `xoxb-` tokens; the agent cannot generate these. This is honest and called out. No drain.

Single-agent, no autosync companion, in a clean working tree. Comment ENG-5416 with the z094 narrative when the slice closes so the loss is in the trail.
2. **Pause parallel batches until autosync's stage-everything behavior is fixed or contained.** Pick one path from z095/z096's open questions (cheapest is (c) `dx wait-for-clean-tree` + serialize commits when sibling agents are mid-commit; principled is range-scoped pre-push + `git stash --include-untracked` before any reset). The agents are spending real time on collision recovery; that time exceeds the time saved by fanning out. Recommend tikur this — three separate failure modes inside one batch is the criterion in `.claude/skills/tikur/`.
3. **Backfill a Linear comment on ENG-5416** with the actual state ("server actions landed; UI was wiped per z094; tests pending; slice not closed"). Treat any Gin-orchestrator-claimed "✓" against an issue without a sibling Linear comment as suspect — FOR-TOM.md says C3 is "in flight" but the orchestrator's running checklist will drift.
